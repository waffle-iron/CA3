DESCRIPTION OF THE PROJECT

We have chosen the first choice of having our own project for this CA and a starting point for our semester project. We have taken the opportunity to work on a real life project.

As part of the community in Nørrebro, Vintro has created a map of local business stores and call it, "Nørrebro On-the-ground". It has started as a business community in Nørrebro with over 300 shops and printed 25,000 cards that are distributed in the east and west, 4 newspapers printed this "map" ...and since then, "app" was sought by many, but have never been developed.

We aim to develop a web application that will be very useful in line with this idea and make use of what we have learned from this semester and nurture our skills while encountering a bunch of error problems. :)

Usage of angular filters




Fetching data from google places API

Since we are not able to make google API requests directly from the client we have decided to make the request from the client to our server, from over server to google, then send the result back to the client. It is not the most optimal way of doing it, because we load more traffic onto our server and there is even more steps the information have to travel through to get to the end user, which also makes the application seeming slow.


We have implemented the use of packages in our program to encapsulate different responsibilities on which parts of the program. It can visibly explain what each does which is extremely important to understand the structure of the program.

TEST DOCUMENTATION

Unit Test Testing - Facade


We are using an in-memory Derby mock database for the JUnit tests. There are only two facade test so far, the reasoning for this is simple, we fetch data directly from google map API. The information is not stored in the DB in the current version of the app, the resulting JSoN is formatted and displayed directly on the client side using AngularJS. So only two simple test were written to check whether the facade methods are working as they should. The createShopTest adds some users and asserts if they are persisted.

REST-ASSURED TEST


We have implemented REST_Assured framework to test the rest services in our project. It was already setup from the seed wherein it initially tests the authentication for the users. We have added some test cases wherein it will validate the response that we get once we have created a new shop and get all the shops in a given path from the use of ShopRest. From there, we are expecting a status code of 200. We also added some tests for the error handling by checking the status code whenever an invalid path was given wherein we get an error code of 404 which means the requested resource was not found.

Manual Testing with Postman


To manually test our api we are using postman with a given path. For example, http://localhost:8080/seedMaven/api/shop/all which returns all the shops as json objects and http://localhost:8080/seedMaven/api/shop which could take a new shop from json that corresponds to the Shop entity class and persist into the database.


HOW TO TEST THE SYSTEM?

This web application is very simple to use. It was designed to display all the shops around Nørrebro. User that has the user role have the privilege of creating/registering new shops. Ideally, the shop owners have this kind of role. And user who has admin role has been granted for all the privileges.

You can login as:
-username: "user" password: "test"
-username: "admin" password: "test"

NOTE: As of now, we only have two shops that has right google shop id: VINTRO and Temple Bar. When you click the name of the shop, it will appear a pop-up that displays the information such as the shop's name, category, address, website that links to their web page, opening hours and contact details. When you have tried to add a new shop as a user role, it will be added to the list and to the database respectively.
